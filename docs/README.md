# Cirrus Design Documents

Cirrus はセルフサービス型プライベートIaaSプラットフォームです。
物理的なハードウェアを抽象化し、仮想的なデータセンターをユーザに提供します。

## 基本思想

IaaSは物理的なハードウェアを抽象化し、仮想的なデータセンターをユーザに提供する。ファシリティや故障、EOLといったハードウェアの実態、FWやハイパーバイザーのアップデートや障害、そういった低レイヤーの面倒くさいところを隠蔽し、ユーザが本当に欲しい機能だけを提供する。

Cirrusの本質は**判断エンジンとオーケストレータ**である。物理の直接制御やファシリティ管理は外部システムに委ねる。

### 宣言的トポロジモデル

物理トポロジをCirrusが直接制御するのではなく、「何が何に到達可能か」を宣言的に登録する。Cirrusはその宣言に基づいて判断を行い、宣言と現実の乖離をヘルスチェックで検出してアラートする。

このパターンは以下の全てに一貫して適用される:

- 到達性ドメイン（ストレージ、ネットワーク）
- ホストのcapability
- 障害トポロジ（ロケーション）
- ホストプロファイル

乖離検出の方向は双方向である。「宣言されているのに到達できない」は障害アラート。「宣言されていないのに到達できる」はセキュリティ・整合性アラート。

### 外部システム連携の原則

Cirrusは判断とオーケストレーションに集中し、物理インフラ管理の実行は外部に委譲する。

- **物理設定の適用** — hookでAWX等を呼び出し、AWXがAnsible/Terraformを実行
- **障害トポロジ** — NetBox等の外部CMDBから同期アダプタ経由でインポート
- **ベアメタルプロビジョニング** — MAAS/Tinkerbell等に委譲、Cirrus側は状態追跡のみ
- **認証** — 外部IdP（Keycloak/Okta等）とOIDC連携

仮想化層の制御（VM操作、OVSデータプレーン制御、ボリューム操作）はCirrusが直接行う。

hookの定義には以下を含む:

- 実行先（AWXのエンドポイントとジョブテンプレートID）
- パラメータマッピング（Cirrus内部データ → AWX変数）
- 完了判定（ポーリングまたはコールバック）
- 失敗時の振る舞い（リトライ回数、ロールバック用ジョブ、エスカレーション先）

## 概念間の関係

### ホストに関する4つの問い

| 問い | 概念 | 導出元 |
|------|------|--------|
| どこに置けるか | コンピュートプール | ストレージドメイン ∩ ネットワークドメイン |
| 何ができるか | Capability | ホストのハードウェア宣言 |
| どう動いているべきか | ホストプロファイル | Capabilityベースのグループに適用 |
| どう運用するか | フォルトドメイン、ロールアウト | ロケーションツリー + ポリシー |

依存関係は上から下に流れる。スケジューラもこの順で候補を絞り込む。

### 管理の単位と到達性

| リソース | ドメイン（管理単位） | 到達性（データプレーン） | 延伸手段 |
|----------|----------------------|--------------------------|----------|
| ストレージ | ストレージバックエンド | レプリケーション範囲 | レプリカ（フルコピー） |
| ネットワーク | — | Geneveオーバーレイ | L3到達性があれば通信可能 |
| コンピュート | コンピュートプール | 拡張コンピュートプール | 拠点間ライブマイグレーション |

全てにおいて「ドメイン ⊆ 到達性」。ドメイン内は高速・低リスク。ドメイン跨ぎは低速・高リスクで明示的な操作。

### スケジューラの統一

初回配置、HA failover、DRS、ストレージマイグレーションは全て同一のスケジューラインフラの上に乗る。

```
[トリガー]        [スケジューラ]         [実行]
初回配置    →                        → VM作成
HA failover →   到達性フィルタ       → ライブマイグレーション
DRS         →   Capabilityマッチ     → ライブマイグレーション
ドレイン    →   プロファイルチェック   → ストレージマイグレーション
EOL対応     →   ロケーション制約     → 計画的移行
            →   スコアリング
            →   最終決定
```

### ライフサイクルの対比

| フェーズ | ホスト | ストレージバックエンド |
|----------|--------|------------------------|
| 導入 | 登録→プロビジョニング→検証→稼働 | 登録→検証→稼働 |
| 通常運用 | DRS、プロファイル適用 | ボリューム作成・削除、QoS |
| 縮退 | — | 新規配置の優先度低下 |
| ドレイン | VMライブマイグレーション（数時間） | ボリューム移行（数日〜数ヶ月） |
| 退役 | 電源OFF | 読み取り専用→退役 |

## 実装の優先順位

### Phase 1: 最小構成

- 少数のストレージバックエンド
- ブロックストレージのみ
- 固定RBAC
- 基本的なVM CRUD + ネットワーク + ボリューム
- ホストのcapability-based placement
- Network/Group/Policy、Egress/Ingress、DNS、DHCP

### Phase 2: 運用機能

- ライブマイグレーション
- DRS
- ホストプロファイル + hookによる適用
- ストレージのドレイン + ライブマイグレーション
- スナップショット・クローンの依存関係管理
- テンプレートサービス + キャッシュコピー
- 監視・メトリクス

### Phase 3: マルチドメイン

- 複数ストレージドメイン + レプリケーション
- 拡張コンピュートプール（拠点間マイグレーション）
- Service Insertion、ゲートウェイHA、テナント間サービス公開
- 外部IPAM連携

### Phase 4: 追加機能

- ファイルストレージ
- オブジェクトストレージ連携
- 帯域管理・QoS
- L4 Load Balancer
- 外部CMDB同期
